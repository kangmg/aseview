## OVERVIEW
aseview is a Python package and CLI for rendering ASE `Atoms` data into
self-contained Three.js HTML viewers. The repo also ships a browser JS module,
MkDocs documentation, generated example viewers, and a VS Code custom editor
extension that reuses the same viewer surface.

## STRUCTURE
```
aseview/
  aseview/              # Python package, runtime templates, themes, static JS
  vscode-extension/     # VS Code custom editor using ase-ts Node workers
  docs/                 # MkDocs source plus generated viewer examples
  tests/                # Python, template, and JS syntax regression tests
  .github/workflows/    # CI, docs deploy, PyPI, VSIX, bundle sync
  pyproject.toml        # Python packaging and console scripts
```

## CONVENTIONS
- Viewer-frame schema must stay aligned across Python and VS Code:
  `symbols`, `positions`, optional `cell`, `pbc`, `forces`, `charges`,
  `magmoms`, `fixed`, `arrays`, `energy`, `name`.
- Python viewers inline packaged vendor JS for notebook/offline HTML. Treat
  `aseview/static/js/vendor/` as runtime assets.
- `aseview/themes/<theme>/` should contain the four viewer templates:
  `molecular_viewer.html`, `normal_viewer.html`, `overlay_viewer.html`,
  `frag_selector.html`.
- `aseview/templates/` is the backward-compatible fallback path. Keep it usable
  even when adding or changing themes.
- VS Code media copies are generated from package templates/styles. Source
  changes to `aseview/templates/molecular_viewer.html`,
  `aseview/static/js/styles.js`, or `vscode-extension/scripts/bundle-viewer.cjs`
  require bundle sync.
- Docs viewer HTML under `docs/assets/viewers/` is generated by
  `docs/generate_examples.py` and intentionally committed for GitHub Pages.
- Release versions normally move together across `pyproject.toml`,
  `aseview/static/js/aseview.js`, `vscode-extension/package.json`, and the
  package-lock root version.
- `.gitignore` ignores broad sample/output globs such as `*.xyz`, `*.cif`,
  `*.json`, and `*.html` with allowlists. New sample data may need an allowlist
  update or explicit force-add.

## ANTI-PATTERNS (THIS PROJECT)
- Do not treat HTML templates as static documentation. They are runtime code
  consumed by Python, browser JS, docs, and VS Code.
- Do not change postMessage names or global template functions casually:
  `viewerLoaded`, `setSettings`, `setData`, `setMolecularData`,
  `initNormalModeViewer`, and copy/save messages bridge multiple surfaces.
- Do not edit only one themed template when the behavior is common to all
  themes. Mirror behavior across base and themed templates or justify the
  theme-specific difference.
- Do not bypass `MolecularData.from_atoms()` for new Python viewer features
  unless the VS Code `ase-ts` path is also considered.
- Do not rely on `--kill` as portable port cleanup. Its implementation reads
  Linux `/proc`.
- Do not create or push `v*` tags until local Python and VS Code release checks
  are done. PyPI publishing is tag/release-driven.

## NOTES
- Package data in `pyproject.toml` must include themes, fallback templates,
  static JS, and vendor JS or generated viewer HTML will miss dependencies.
- `MANIFEST.in` is narrower than `pyproject.toml` package-data. Check both when
  packaging assets change.
- CI checks Python 3.9 through 3.12, JS syntax for two package JS files, and
  flake8 error classes only.
- The bundle sync workflow can auto-commit generated VS Code media after source
  template/style changes.
- Full browser and VS Code end-to-end behavior is not covered by the Python test
  suite; template edits deserve manual visual checks when UI behavior matters.
- Viewer projection defaults to `Perspective`; `Orthographic` remains available
  as an explicit selectable mode.
- Local ignored artifacts may be present (`.venv/`, `.claude/worktrees/`,
  `site/`, `dist/`, `build/`, `vscode-extension/node_modules/`). Exclude them
  from structure scans unless the task is explicitly about generated output.
